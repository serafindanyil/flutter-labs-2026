# Лабораторна робота №3: Додавання логіки та роботи з локальним сховищем

Цей гайд допоможе тобі, як React Native розробнику, легко розібратися зі змінами у цій лабораторній роботі та впевнено захистити її.

## 1. Локальне сховище: `flutter_secure_storage` vs `AsyncStorage`

У React Native ти, ймовірно, використовував `AsyncStorage` (або `Expo SecureStore`). У Flutter його аналогом є плагіни `shared_preferences` (для звичайних даних) або `flutter_secure_storage` (для захищених даних). 
Згідно з порадами до лабораторної, ми використали `flutter_secure_storage`.
Він зберігає дані у Keychain (на iOS) та Keystore (на Android).

```dart
// React Native (AsyncStorage)
// await AsyncStorage.setItem('key', 'value');

// Flutter (flutter_secure_storage)
// await _storage.write(key: 'key', value: 'value');
```

## 2. Віджети зі станом: `StatefulWidget` vs `Class / Functional Components with State`

Ми використали `StatefulWidget` на сторінках авторизації та налаштувань. У Flutter `StatefulWidget` — це аналог компонента з локальним стейтом у React (наприклад, через `useState` або `this.state`).

- `setState(() { ... })` працює так само як і в React — повідомляє фреймворку, що стан змінився і потрібно перемалювати (виклик `build()`) віджет.
- Контролери замінюють керовані (controlled) інпути. Замість того, щоб на кожен натиск клавіші зберігати `onChangeText` у стейт, ми передаємо `TextEditingController` у `TextField`, і він сам рулить текстом, а ми просто зчитуємо `_controller.text`, коли сабмітимо форму.

### 2.1 Кастомний віджет пароля (`PasswordTextField`)
Ми також створили окремий `StatefulWidget` — `PasswordTextField`. У ньому створено внутрішній стан `_obscured`, що перемикається при натисканні на іконку очка (`LucideIcons.eye` / `LucideIcons.eyeClosed`). Це аналог створення окремого компонента в React Native з `useState(true)` для властивості `secureTextEntry`.

### 2.2 Кастомний компонент Toast (`AppToast`)
Замість стандартного `SnackBar`, ми  з'ясували, що проекту потрібна більш красива уніфікована система повідомлень. Тому створили клас `AppToast` з двома статичними методами:
- `AppToast.error(context, message)` — червоний toast з іконкою `LucideIcons.triangleAlert` для помилок
- `AppToast.success(context, message)` — зелений toast з іконкою `LucideIcons.check` для успіху

```dart
// React Native: звичайно користувалися Alert.alert() або Toast.show()
// Toast.show({ type: 'success', text1: 'Success!' });

// Flutter: тепер використовуємо AppToast
// AppToast.success(context, 'Успішний вхід!');
```

Кожен toast має:
- Кастомні кольори з дизайн-системи (`AppColors.danger`, `AppColors.success`)
- Іконку з дизайн-системи (Lucide Icons)
- Floating поведінку (поява знизу екрана, не витісняючи контент)
- Скруглені кути та цільову іконку для кращого UX

Ми використовуємо глобальні Toast-повідомлення (`AppToast.success(...)`) для сповіщення користувача про успішні важливі події життєвого циклу додатка: після успішного входу, реєстрації, збереження налаштувань профілю або виходу з системи. Це покращує зворотній зв'язок з користувачем — типова найкраща практика як у React Native, так і у Flutter застосунках.

## 3. Навігація

У цій лабі ми реалізували переходи між сторінками за допомогою `Navigator`. Це працює подібно до `React Navigation` (стек-навігація).
- `Navigator.push` — це `navigation.navigate()` у React Native.
- `Navigator.pushAndRemoveUntil(..., (route) => false)` — це очищення стеку і встановлення нового екрана (схоже на `navigation.reset()`). Це корисно, щоб після логіну користувач не міг кнопкою "Назад" повернутись на екран вводу пароля.

## 4. Архітектура: Репозиторії та Абстракції

Ми реалізували патерн Repository для роботи з даними. Це було однією з вимог (щоб легко було підключити справжнє API у майбутньому).

Ми створили абстрактний інтерфейс `AuthRepository` (це ніби `.d.ts` тип для сервісу) та його імплементацію для локального сховища `AuthRepositoryLocal`.

```dart
// Абстракція потрібна для того, щоб якщо колись з'явиться Firebase/Backend,
// ми просто створили AuthRepositoryApi і змінили ініціалізацію в одному місці.
abstract interface class AuthRepository {
  Future<void> register(UserModel user);
  Future<UserModel?> login(String email, String password);
// ...
}
```

Для "ін'єкції залежностей" (DI), щоб дотриматись правила про модифікатори `final` чи `const` та уникнути глобальних змінних з можливістю мутації, ми використали `abstract final class AuthProvider` зі статичною фінальною зміною (у Flutter немає `export const object`, але це його найближчий аналог для простого зберігання залежностей у singleton).

## 5. Валідація: Електронна пошта, цифри в імені та перевірка паролів

Ми додали просту валідацію через `RegExp` та звичайні умови:
- Для e-mail перевіряємо формат: `RegExp(AppConstants.emailRegex).hasMatch(email)`
- Для імені перевіряємо на відсутність цифр: `RegExp(r'\d').hasMatch(name)`
- При реєстрації перевіряємо збіг введених паролів у двох полях: `password == confirmPassword`. Якщо вони не збігаються, у полі "Повторіть пароль" показується локальна помилка "Паролі не співпадають", зберігаючи водночас повідомлення з вимогами до довжини у головному полі пароля.

Додатково ми перевіряємо існування акаунту ще на етапі вводу емейлу (до переходу на сторінку пароля). Для цього додано метод `checkEmailExists` у `AuthRepository`. Якщо користувач в логіні вводить неіснуючий емейл (або при реєстрації вже існуючий), ми не пускаємо його далі і показуємо глобальну помилку-сповіщення (Toast), викликаючи `AppToast.error(context, ...)`.

У разі локальних помилок вводу (наприклад, пусте поле, занадто короткий пароль чи невірний формат e-mail) ми робимо `setState` для стрингової змінної помилки (наприклад, `_emailErrorText`), яку `PrimaryTextField` розпізнає та малює червоним текстом. Щоб додавання помилки не міняло висоту інпута та не здвигало інші елементи на екрані (layout shift), віджет помилки розміщено у `Stack` з позиціонуванням `Positioned` із виходом за нижній край `bottom: -22` (це як `position: absolute` у React Native / CSS). Текст плавно з'являється завдяки `AnimatedOpacity`. Бізнес-помилки типу «Невірний пароль» або «Емейл вже зареєстровано» також тепер показуються тільки через `AppToast`.

## 6. Демонстрація (Що треба показати на відео)

Для здачі лаби не забудь зняти коротке відео. Покажи:
1. Як ти вводиш ім'я (з цифрами) і емейл без `@` — покажи повідомлення про помилку лінтера.
2. Успішну реєстрацію (перехід на екран пароля і далі).
3. Після цього перейди на екран налаштувань (Settings / Профіль).
4. Зміни своє ім'я, клікнувши на іконку олівця. Покажи, що немає окремої кнопки "Зберегти" — зміни зберігаються автоматично після натискання клавіші Enter (submit) або коли ти клікаєш поза полем вводу (знімаєш фокус). З'явиться Toast-сповіщення "Зміни збережено!".
5. Натисни розлогінитись, щоб повернутися на сторінку входу.
6. Введи якісь рандомні неправильні дані на сторінці входу, щоб показати валідацію повідомленням "Невірний емейл або пароль".
7. Увійди у створений акаунт для перевірки логіну.

## 7. Відповіді на типові питання викладача

**Питання:** Навіщо ви створили `AuthRepository` (інтерфейс) окремо від класу реалізації?
**Відповідь:** Згідно з Dependency Inversion Principle паттерну SOLID. Якщо у подальшому ми змінимо Local Storage на сервер через REST API, ми залишимо `abstract interface class AuthRepository` як є, а створимо просто новий клас, і нам не доведеться змінювати код всередині екранів UI.

**Питання:** Як ви робите перемальовку екрану після зміни даних (наприклад, змінили ім'я у профілі)?
**Відповідь:** Ми викликаємо метод `setState()`, куди передаємо оновлені дані. Flutter розуміє, що стан віджета `StatefulWidget` змінився, і викликає метод `build()`, де UI будується заново з новим `_user!.name` або відображує інпути для режиму `_isEditing`.

**Питання:** Чому обрали `flutter_secure_storage`?
**Відповідь:** Бо він більш захищений за звичайний `SharedPreferences`. На iOS він використовує Keychain, що дозволяє безпечно зберігати інформацію авторизації.

**Питання:** Як ви реалізували кастомну систему NotificationToast(`AppToast`) та вирішили проблему спаму однакових повідомлень?
**Відповідь:** Ми додали статичні методи `error()` та `success()`. 
- Щоб вирішити проблему, коли користувач агресивно "спамить" сабміт форму (багатократно викликаючи той самий Toast), я додав просту реалізацію **тротлінгу (Throttling)**. Ми запам'ятовуємо останнє повідомлення (`_lastMessage`) і час (`_lastTime`). Якщо однакове повідомлення викликається поки попереднє ще не зникло з екрана — ми його ігноруємо. Це розв'язує проблему, коли однакові Toast'и накладуються один на одний.
- Щоб домогтися ідеальної плавної анімації "зсуву вниз" наче в нативних Notification, ми замінили стандартний `SnackBar` на пакет `toastification`. Ми використали кастомний `animationBuilder` з `SlideTransition`, який анімує зміщення з `Offset(0, 1)` (знизу) до `Offset(0, 0)` при появі, та плавно ховає тост донизу при зникненні.
- Додатково в налаштуваннях `toastification` додано свайп щоб закрити Toast жестом донизу: `dragToClose: true` та напрямок `DismissDirection.down`.
- Також, щоб уникнути "мигання" (layout jumping) кнопок при появі Toast та початку завантаження, ми більше не замінюємо віджет кнопки на `CircularProgressIndicator`. Замість цього, ми передаємо параметр `isLoading` прямо у `PrimaryButton`, який сам обробляє стан завантаження, зберігаючи свій розмір на екрані. Це аналог передачі `loading={true}` у кастомний компонент кнопки в React Native.

Удачі на здачі!

**Питання:** Як ви реалізували авто-збереження профілю (без кнопки «Зберегти») при знятті фокуса з інпута?
**Відповідь:** У React Native ми б використовували `onBlur` пропсу інпута. У Flutter ми додали параметр `focusNode` у наш кастомний `PrimaryTextField` і передали йому екземпляр `FocusNode()` зі сторінки налаштувань. Ми підписалися на зміну фокуса через `_nameFocusNode.addListener(...)`. Якщо користувач завершує редагування і фокус втрачається (`!_nameFocusNode.hasFocus`), ми автоматично викликаємо `_saveChanges()`. Для натискання `Enter` на клавіатурі ми обробляємо колбек `onFieldSubmitted` (еквівалент `onSubmitEditing` в RN). Крім того, весь екран обгорнуто в `GestureDetector`, який при тапі в поза полем викликає `FocusScope.of(context).unfocus()`, знімаючи фокус і тригерючи збереження.
